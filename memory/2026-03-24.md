### 系统概述
4 人代码评审团队，由 4 个独立 cron 任务组成，每个角色使用不同模型：

| 角色 | Cron Job ID | 模型 | 职责 |
|------|-------------|------|------|
| 设计师 | `a0ea64d4-4118-43e3-9cf3-25dc84909038` | `qwen3-coder-plus` | 代码架构设计、初版实现 |
| 校核员 | `d10df21e-7512-4751-93c6-0901e51cf632` | `qwen3.5-plus` | 代码质量检查、风格审查 |
| 审核员 | `b0006971-6353-49e6-9892-354ec527651d` | `minimax-m2.5` ⭐ | 安全性、最佳实践审查 |
| 总工 | `41af46d0-9f18-4f60-b31a-8f5ca02ffe9b` | `glm-5` | 最终审批、合并决策 |

### 工作流程
```
用户请求 → 设计师 → 校核员 → 审核员 → 总工 → 合并到 plugins/
```

### 文件位置
- **协调器脚本**: `scripts/code-review-coordinator.ps1`
- **使用文档**: `docs/code-review-team.md`
- **任务目录**: `code-review/{task-id}/`
- **交付位置**: `D:\OpenClaw\plugins\`

### 使用方法
**推荐方式**：在 Telegram 中说"生成 XX 插件"或"创建 XX 代码"

**手动方式**：
```powershell
cd D:\OpenClaw\.openclaw\workspace\scripts
.\code-review-coordinator.ps1 start -Request "生成一个天气查询插件"
```

### 触发机制
- 4 个 cron 任务都设置为一次性任务（2026-12-31）
- 实际通过 `openclaw cron run --job-id <id>` 手动触发
- 每个角色完成后创建 `.done` 标记文件
- 协调器脚本负责检测标记并触发下一阶段

### 注意事项
1. 审核员使用 MiniMax-M2.5（用户特别要求）
2. 所有代码最终交付到 `D:\OpenClaw\plugins\`
3. 每个任务有独立的评审报告
4. 总工负责最终审批和合并决策
