# 灾难恢复与极端情况演练规格

## 一、目标

系统必须具备：
- ✅ 能识别异常（网络/API/磁盘/内存）
- ✅ 能自动进入保护模式
- ✅ 能避免错误扩散（防止雪崩）
- ✅ 能在条件恢复后自动回到正常状态
- ✅ 每周至少执行一次模拟演练

## 二、状态机设计

### 系统状态

| 状态 | 说明 | 允许的操作 |
|------|------|-----------|
| 🟢 NORMAL | 正常模式 | 所有任务 |
| 🟡 DEGRADED | 降载模式 | 核心 + 监控任务 |
| 🔴 SAFE | 安全模式 | 仅核心任务 |
| ⚫ OFFLINE | 离线模式 | 仅本地任务 |
| 🔄 RECOVERING | 恢复中 | 分阶段恢复 |

### 状态转换

```
NORMAL ──检测异常──> DEGRADED/SAFE/OFFLINE
   ↑                      ↓
   └────恢复验证──── RECOVERING
```

## 三、极端场景与应对策略

### 场景 1: API 不可用 🚫

**识别条件**:
- 连续 API 调用失败 ≥ 5 次
- 或 5 分钟内失败率 > 50%

**应对策略**:
1. 暂停所有依赖 API 的非关键任务
2. 启用 fallback 模型（glm-5）
3. 降低请求频率（至少降低 50%）
4. 避免 auto_healer 高频 retry
5. 记录事件到 disaster-log.md

**恢复条件**:
- 连续 3 次 API 调用成功
- 自动恢复正常模式

### 场景 2: 网络中断 🌐

**识别条件**:
- 无法访问外部服务（HTTP 全失败）
- 测试 URL: google.com, github.com, api.github.com

**应对策略**:
1. 进入 OFFLINE 模式
2. 暂停：网站监控、邮件监控、外部 API 任务
3. 保留：本地日志清理、配置检查、自检任务
4. 缓存待发送消息（恢复后补发）

**恢复条件**:
- 连续 2 次网络检测成功
- 自动进入 RECOVERING → NORMAL

### 场景 3: 磁盘满 💾

**识别条件**:
- 磁盘使用 > 95%

**应对策略** (按顺序执行):
1. 删除 30 天前日志
2. 压缩 7 天前日志
3. 删除非关键缓存
4. 暂停 backup、日志写入密集任务
5. 若仍 > 95%：进入安全模式 + 发送紧急告警

**恢复条件**:
- 磁盘使用 < 85%
- 自动进入 RECOVERING → NORMAL

### 场景 4: 内存爆满 🧠

**识别条件**:
- 内存使用 > 95%

**应对策略**:
1. 触发垃圾回收 `[System.GC]::Collect()`
2. 清理缓存
3. 停止高内存任务
4. 若仍未恢复：重启 gateway
5. 若重启失败：进入安全模式

**恢复条件**:
- 内存使用 < 85%
- 自动进入 RECOVERING → NORMAL

### 场景 5: 任务雪崩 ❄️

**识别条件**:
- 10 分钟内失败任务 ≥ 5
- auto_healer 频繁触发

**应对策略**:
1. 暂停 auto_healer 10 分钟（防止风暴）
2. 暂停低优先级任务
3. 限制任务执行速率
4. 逐个恢复任务（而不是全部同时启动）

**恢复条件**:
- 失败任务数 < 3
- auto_healer 暂停时间到期
- 分阶段恢复任务

## 四、演练机制

### 每周灾难演练

**Agent**: 🧪 灾难演练员

**频率**: 每周日 10:00

**职责**:
1. 随机选择一个场景
2. 模拟异常条件
3. 验证系统是否正确检测
4. 验证应对策略是否执行
5. 验证恢复能力
6. 输出演练报告

**报告位置**: `memory/disaster-drill-report.md`

### 演练场景池

| 场景 | 模拟方式 |
|------|---------|
| API 失败 | 设置 apiFailureCount = 5 |
| 网络中断 | 逻辑模拟（无法完全模拟） |
| 磁盘满 | 逻辑模拟（无法完全模拟） |
| 内存满 | 逻辑模拟（无法完全模拟） |
| 任务雪崩 | 逻辑模拟（无法完全模拟） |

## 五、自动恢复机制

### 恢复流程

```
异常状态 (DEGRADED/SAFE/OFFLINE)
         ↓
  检查恢复条件
         ↓
    满足条件？
    /       \
  是         否
  ↓          ↓
RECOVERING  继续等待
  ↓
阶段 1: 恢复核心任务 (5s)
  ↓
阶段 2: 恢复中优先级任务 (5s)
  ↓
阶段 3: 恢复低优先级任务 (5s)
  ↓
NORMAL
```

### 恢复条件汇总

| 原状态 | 恢复条件 |
|--------|---------|
| DEGRADED | API 连续 3 次成功 |
| OFFLINE | 网络连续 2 次检测成功 |
| SAFE | 内存 < 85% 且 磁盘 < 85% |

## 六、日志与审计

### 日志文件

| 文件 | 用途 |
|------|------|
| `memory/disaster-log.md` | 所有异常事件记录 |
| `memory/disaster-drill-report.md` | 每周演练报告 |
| `memory/disaster-recovery-state.json` | 状态机当前状态 |

### 事件记录格式

```
[YYYY-MM-DD HH:mm:ss] [LEVEL] message | key=value
```

**级别**:
- INFO: 常规信息
- WARNING: 警告事件
- ERROR: 错误事件
- CRITICAL: 严重事件
- STATE_CHANGE: 状态变更
- DRILL: 演练事件

## 七、最终保护机制

当系统无法恢复时：

1. **停止所有任务**
   - 防止错误扩散
   - 保留系统状态

2. **保留日志**
   - 所有事件记录完整保存
   - 便于事后分析

3. **等待人工干预**
   - 发送紧急通知
   - 提供详细诊断信息

## 十、监控指标

| 指标 | 阈值 | 告警级别 | 动作 |
|------|------|---------|------|
| API 连续失败 | ≥ 5 | 🔴 紧急 | 进入 DEGRADED |
| 网络测试失败 | 100% | 🔴 紧急 | 进入 OFFLINE |
| 磁盘使用率 | > 95% | 🔴 紧急 | 清理 → SAFE |
| 内存使用率 | > 95% | 🔴 紧急 | 清理 → 重启 → SAFE |
| 失败任务数 (10min) | ≥ 5 | 🔴 紧急 | 暂停 auto_healer |
| auto_healer 修复 (10min) | ≥ 3 | 🟡 警告 | 暂停 10 分钟 |
