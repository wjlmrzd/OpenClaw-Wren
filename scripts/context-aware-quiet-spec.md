# 🔕 情境感知静默 (Context-Aware Quiet)

**职责：** 智能调整通知策略，避免打扰用户

---

## 核心概念

传统静默规则：
```
22:00-06:00 → 仅紧急通知
```

情境感知静默：
```
时间 + 用户状态 + 事件严重性 + 历史行为 → 智能决策
```

---

## 情境维度

### 1. 时间情境

| 时段 | 名称 | 通知策略 |
|------|------|---------|
| 06:00-09:00 | 晨间 | 正常通知（可发送早报/日历） |
| 09:00-12:00 | 工作上午 | 正常通知（工作相关优先） |
| 12:00-14:00 | 午休 | 降级通知（非紧急延迟） |
| 14:00-18:00 | 工作下午 | 正常通知 |
| 18:00-22:00 | 晚间 | 正常通知（可发送总结） |
| 22:00-06:00 | 深夜 | 静默（仅紧急） |

### 2. 用户状态情境

**检测方式：**
- 最后活跃时间（Telegram 最后上线）
- 最后回复时间
- 日历状态（忙碌/空闲/会议中）
- 系统活动（可选，高级）

**状态推断：**

| 状态 | 判断条件 | 通知策略 |
|------|---------|---------|
| 🟢 活跃 | 5 分钟内有互动 | 正常通知 |
| 🟡 离开 | 5-30 分钟无互动 | 降级通知（累积发送） |
| 🟠 忙碌 | 日历显示会议中 | 仅紧急通知 |
| 🔴 勿扰 | 22:00-06:00 或手动设置 | 静默（仅危机） |

### 3. 事件严重性

| 级别 | 定义 | 通知条件 |
|------|------|---------|
| 🟢 信息 | 日常报告、完成通知 | 仅活跃时段 + 用户活跃 |
| 🟡 注意 | 警告、非紧急问题 | 活跃时段 + (活跃或离开) |
| ⚠️ 警告 | 需要关注的问题 | 所有时段（除勿扰） |
| 🔴 紧急 | 系统故障、数据丢失风险 | 所有时段（包括勿扰） |
| 🔥 危机 | 立即干预，否则严重后果 | 无条件通知 |

### 4. 历史行为学习

**学习内容：**
- 用户对不同类型通知的响应率
- 用户偏好的通知时段
- 用户标记为"打扰"的通知类型

**示例：**
```
用户经常在 23:00 后回复消息
→ 推断：夜猫子，深夜静默阈值可放宽

用户对"任务失败"通知响应率高
→ 优先级：任务失败 > 资源警告

用户从不查看"周报"类通知
→ 降级：周报改为每周摘要
```

## 决策引擎

### 决策流程

```
事件发生
    ↓
判断事件严重性
    ↓
获取当前情境（时间 + 用户状态）
    ↓
查询历史偏好
    ↓
计算通知优先级
    ↓
决策：立即通知 / 延迟通知 / 静默
    ↓
执行
```

### 决策矩阵

| 严重性 \ 情境 | 活跃 | 离开 | 忙碌 | 勿扰 |
|-------------|------|------|------|------|
| 🟢 信息 | ✅ 立即 | ⏳ 累积 | ⏳ 累积 | ❌ 静默 |
| 🟡 注意 | ✅ 立即 | ⏳ 累积 | ⏳ 累积 | ❌ 静默 |
| ⚠️ 警告 | ✅ 立即 | ✅ 立即 | ⏳ 延迟 | ⏳ 延迟 |
| 🔴 紧急 | ✅ 立即 | ✅ 立即 | ✅ 立即 | ✅ 立即 |
| 🔥 危机 | ✅ 立即 | ✅ 立即 | ✅ 立即 | ✅ 立即 + 电话 |

### 优先级计算

```python
priority_score = (
    severity_weight * 40 +      # 严重性权重 40%
    user_state_weight * 30 +    # 用户状态权重 30%
    time_weight * 20 +          # 时间权重 20%
    history_weight * 10         # 历史权重 10%
)

if priority_score > 80:
    return "立即通知"
elif priority_score > 50:
    return "延迟通知 (30 分钟后)"
else:
    return "静默 (累积到摘要)"
```

## 实现方案

### 方案 1: 轻量级（推荐）

**实现方式：** 修改现有 Agent 的通知逻辑

**优点：**
- 无需新增 Agent
- 侵入性小
- 易于维护

**缺点：**
- 功能分散
- 难以统一管理

**实施：**
在每个 Agent 的提示词中添加静默规则：
```
发送 Telegram 前检查：
1. 当前时间是否在静默时段 (22:00-06:00)
2. 事件严重性是否 >= 警告级别
3. 如否，延迟到 06:00 发送摘要
```

### 方案 2: 中等复杂度

**实现方式：** 新增通知协调员 Agent

**职责：**
- 统一管理所有通知
- 接收其他 Agent 的通知请求
- 根据情境决策是否/何时发送

**优点：**
- 集中管理
- 易于扩展
- 可学习用户偏好

**缺点：**
- 需要新增 Agent
- 增加系统复杂度

**实施：**
```
Agent → 通知协调员 → 情境判断 → Telegram
```

### 方案 3: 高级（未来）

**实现方式：** 机器学习 + 用户行为分析

**功能：**
- 学习用户作息规律
- 预测最佳通知时间
- 个性化通知策略

**优点：**
- 高度智能
- 越用越准

**缺点：**
- 实现复杂
- 需要数据积累

## 通知累积与摘要

### 累积队列

**队列管理：**
- 静默时段的通知进入累积队列
- 队列按严重性排序
- 队列按来源分组

### 摘要生成

**触发条件：**
- 静默时段结束（06:00）
- 用户状态变为活跃
- 队列累积超过阈值（10 条）

**摘要格式：**
```
📬 静默时段摘要 (06:00)

📊 概览:
- 累积通知：12 条
- 已处理：5 条（自动修复）
- 需关注：3 条
- 信息类：4 条

⚠️ 需关注:
1. 🧪 回归测试员 - 模型验证失败 (已修复)
2. 🚑 故障自愈员 - 任务超时 (已调整)
3. 📡 事件协调员 - 内存警告 (已清理)

ℹ️ 信息类:
- 🧹 日志清理员 - 清理完成
- 🧪 回归测试员 - 测试通过 (x3)
- ...

查看详情：memory/event-log.md
```
