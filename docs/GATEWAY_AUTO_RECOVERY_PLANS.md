# OpenClaw 网关自动修复/重启方案

> 基于 OpenClaw 官方文档和社区最佳实践设计

## 方案一：基础监控 + 手动修复（推荐入门）

**适用场景**: 稳定环境，偶尔需要维护

### 功能
- 每6小时自动健康检查 (`openclaw doctor`)
- 发现问题时发送 Telegram 报告
- 提供一键修复命令

### 实现
```bash
# Cron 任务：每6小时运行健康检查
cron add --schedule "0 */6 * * *" \
  --command "openclaw doctor --json | telegram-report"
```

### 优点
- 简单可靠，不自动操作避免误伤
- 人工确认后再执行修复

### 缺点
- 需要人工介入

## 方案二：自动重启 + 故障转移（推荐生产环境）

**适用场景**: 7x24 运行，要求高可用

### 功能
1. **健康检查**: 每分钟检测 Gateway 状态
2. **自动重启**: 检测到无响应时自动重启
3. **配置修复**: 自动修复常见配置问题
4. **双重保险**: Windows 服务 + Cron 监控

### 实现
```javascript
// 监控脚本逻辑
if (!gatewayHealthy()) {
  // 1. 尝试优雅重启
  exec("openclaw gateway restart");
  
  // 2. 等待恢复
  await sleep(10s);
  
  // 3. 仍未恢复，强制重启服务
  if (!gatewayHealthy()) {
    exec("Restart-Service OpenClawGateway");
  }
  
  // 4. 发送报告
  telegram.send("网关已自动重启，原因: ${reason}");
}
```

### Cron 配置
```json
{
  "name": "gateway-health-monitor",
  "schedule": {"kind": "every", "everyMs": 60000},
  "payload": {
    "kind": "agentTurn",
    "message": "检查 Gateway 健康状态，如异常则自动重启并报告"
  },
  "sessionTarget": "isolated"
}
```

### 优点
- 全自动，无需人工
- 快速恢复（<30秒）

### 缺点
- 需要仔细测试避免循环重启

## 方案三：智能修复 + 预防性维护（推荐高级用户）

**适用场景**: 复杂环境，有多种故障模式

### 功能
1. **预防性维护**: 预测性检测潜在问题
2. **分级响应**:
   - L1: 自动修复（配置错误、权限问题）
   - L2: 自动重启（服务卡死）
   - L3: 告警通知（需要人工）
3. **自我修复学习**: 记录修复历史，优化策略

### 自动修复场景
| 问题 | 自动修复动作 |
|------|-------------|
| 配置 JSON 语法错误 | 回滚到上次备份 |
| Token 过期 | 切换到备用 Token |
| 端口占用 | 自动更换端口 |
| 内存泄漏 | 定时重启 |
| 磁盘空间不足 | 清理日志 |

### 优点
- 最智能，最少人工干预
- 可学习和优化

### 缺点
- 实现复杂
- 需要充分测试
